# Penjelasan Sequence Diagram

Sequence diagram menggambarkan interaksi antar objek dalam sistem secara berurutan berdasarkan waktu. Setiap diagram menggunakan notasi *robustness stereotype* yang terdiri dari: **Boundary Class** (menangani komunikasi antarmuka pengguna), **Control Class** (menangani logika proses), dan **Entity Class** (merepresentasikan data/basis data). Selain itu digunakan notasi *Activation* (durasi aktivasi proses), *Life Line* (garis putus-putus yang menghubungkan objek), dan *Recursive* (pesan ke dirinya sendiri).

---

Pada Gambar 4.X1 ditunjukkan *sequence diagram* untuk proses login. Interaksi dimulai ketika aktor *User* membuka halaman login. Boundary *Halaman Login* merespons dengan menampilkan form yang berisi kolom email dan password. Setelah User mengisi kredensial dan mengirimkan form, boundary meneruskan data ke control *AuthController*. AuthController kemudian meminta *User Model* (entity) untuk memverifikasi kredensial, yang kemudian mengirimkan query validasi ke *Database* (entity). Database mengembalikan hasil validasi ke User Model, yang diteruskan kembali ke AuthController. AuthController selanjutnya melakukan proses *recursive* untuk mengidentifikasi role pengguna (Admin, Guru, Kepala Sekolah, atau Kiosk Presensi). Setelah role teridentifikasi, AuthController menginstruksikan Halaman Login untuk melakukan redirect ke dashboard yang sesuai, dan User melihat halaman dashboard sesuai role-nya.

Pada Gambar 4.X2 ditunjukkan *sequence diagram* untuk proses kelola data guru. Interaksi dimulai ketika aktor *Admin* membuka menu data guru. Boundary *Halaman Guru* mengirimkan request ke control *TeacherController*, yang kemudian meminta *Teacher Model* (entity) untuk mengambil data guru dari *Database* (entity). Database mengembalikan data, dan daftar guru ditampilkan kepada Admin melalui Halaman Guru. Selanjutnya Admin memilih aksi (tambah, edit, atau hapus). Jika memilih tambah atau edit, Halaman Guru menampilkan form isian, Admin mengisi data dan mengirimkan form. TeacherController melakukan proses *recursive* untuk memvalidasi data yang dimasukkan. Jika valid, TeacherController meminta Teacher Model menyimpan data ke Database, dan notifikasi berhasil dikirim kembali ke Admin melalui Halaman Guru. Jika memilih hapus, setelah Admin mengonfirmasi, TeacherController meminta Teacher Model untuk menghapus data dari Database.

Pada Gambar 4.X5 ditunjukkan *sequence diagram* untuk proses kelola kriteria SAW. Interaksi dimulai ketika aktor *Admin* membuka menu kriteria penilaian. Pada Gambar 4.X6 ditunjukkan *sequence diagram* untuk proses generate QR Code absensi. Interaksi dimulai ketika aktor *Admin* membuka menu generate QR Code. Boundary *Halaman QR* mengirimkan request ke control *AttendanceController* untuk mengambil daftar guru. AttendanceController meminta *Teacher Model* (entity) mengambil data guru dari *Database* (entity), kemudian daftar guru ditampilkan kepada Admin. Admin memilih guru yang akan dibuatkan QR Code dan menekan tombol generate. AttendanceController melakukan proses *recursive* pertama untuk mengenkripsi data absensi (teacher_id, nama, tanggal, timestamp) menggunakan algoritma AES-256-CBC, kemudian proses *recursive* kedua untuk men-generate gambar QR Code dalam format SVG dengan masa berlaku yang ditentukan. Hasilnya dikirim kembali ke Halaman QR yang menampilkan QR Code pada layar.

Pada Gambar 4.X7 ditunjukkan *sequence diagram* untuk proses scan QR Code absensi. Interaksi dimulai ketika aktor *Guru* membuka menu scan QR Code pada perangkatnya. Boundary *Halaman Scanner* mengaktifkan kamera dan Guru mengarahkan kamera ke QR Code. Setelah QR Code terbaca, data dikirimkan ke control *AttendanceController*. AttendanceController melakukan tiga proses *recursive* berturut-turut: pertama mendekripsi data QR Code, kedua memvalidasi masa berlaku (apakah belum melewati batas waktu), dan ketiga memvalidasi geolokasi menggunakan formula Haversine (apakah jarak guru dari koordinat sekolah masih dalam radius yang diizinkan). Jika seluruh validasi lolos, AttendanceController meminta *Attendance Model* (entity) untuk memeriksa data absensi hari ini di *Database* (entity). Berdasarkan hasilnya, sistem mencatat check-in (dengan penentuan status tepat waktu atau terlambat) atau check-out, menyimpan data beserta koordinat GPS ke Database, dan mengirimkan notifikasi keberhasilan kembali ke Guru melalui Halaman Scanner.

Pada Gambar 4.X8 ditunjukkan *sequence diagram* untuk proses lihat rekap absensi. Interaksi dimulai ketika aktor *Pengguna* membuka menu rekap absensi. Boundary *Halaman Rekap* mengirimkan request ke control *AttendanceController*, yang merespons dengan menampilkan halaman beserta filter pencarian. Pengguna memilih filter yang diinginkan (periode waktu, nama guru, atau status kehadiran) dan menekan tombol tampilkan. Halaman Rekap mengirimkan request dengan parameter filter ke AttendanceController. AttendanceController meminta *Attendance Model* (entity) untuk mengambil data absensi dari *Database* (entity) sesuai filter. Database mengembalikan data, yang kemudian diteruskan ke Halaman Rekap dalam bentuk tabel rekap absensi yang mencakup tanggal, nama, waktu check-in, check-out, dan status kehadiran.

Pada Gambar 4.X9 ditunjukkan *sequence diagram* untuk proses input nilai siswa. Interaksi dimulai ketika aktor *Guru* membuka menu input nilai. Boundary *Halaman Nilai* mengirimkan request ke control *GradeController*, yang merespons dengan menampilkan form pemilihan kelas dan mata pelajaran. Setelah Guru memilih, GradeController meminta *Grade Model* (entity) mengambil data siswa beserta nilai yang sudah ada (jika ada) dari *Database* (entity), kemudian menampilkan form tabel input nilai. Guru mengisi nilai setiap siswa (tugas harian, UTS, UAS, sikap, keterampilan) dan mengirimkan form. GradeController melakukan dua proses *recursive*: pertama memvalidasi bahwa setiap nilai berupa angka dalam rentang 0–100, dan kedua menghitung nilai akhir dengan formula pembobotan (Tugas Harian × 0,30 + UTS × 0,30 + UAS × 0,40). Jika valid, GradeController meminta Grade Model menyimpan seluruh data nilai ke Database, dan notifikasi berhasil dikirim kembali ke Guru.

Pada Gambar 4.X10 ditunjukkan *sequence diagram* untuk proses lihat ranking siswa menggunakan metode SAW. Interaksi dimulai ketika aktor *Pengguna* membuka menu ranking siswa. Boundary *Halaman SAW* mengirimkan request ke control *SAWController*, yang merespons dengan menampilkan halaman beserta filter (kelas, tahun ajaran). Setelah Pengguna memilih filter dan menekan tombol hitung, SAWController memanggil *SAWService* (entity) untuk menjalankan perhitungan. SAWService mengambil data siswa dan kriteria dari *Database* (entity), lalu menjalankan empat proses *recursive* berturut-turut: (1) membangun matriks keputusan berdasarkan kriteria C1–C4, (2) melakukan normalisasi matriks, (3) menghitung nilai preferensi (skor SAW) dengan menjumlahkan perkalian bobot dan nilai normalisasi, dan (4) melakukan perankingan dari skor tertinggi ke terendah. Hasil ranking dikembalikan ke SAWController, yang menyimpannya ke Database dan mengirimkan tabel ranking beserta detail perhitungan SAW ke Halaman SAW untuk ditampilkan kepada Pengguna.

Pada Gambar 4.X12 ditunjukkan *sequence diagram* untuk proses export laporan. Interaksi dimulai ketika aktor *Pengguna* membuka menu laporan. Boundary *Halaman Laporan* mengirimkan request ke control *ReportController*, yang merespons dengan menampilkan halaman beserta filter dan opsi format. Pengguna memilih jenis laporan, mengatur filter (periode, kelas, status), memilih format export (Excel atau PDF), dan menekan tombol export. ReportController melakukan proses *recursive* untuk memeriksa format yang dipilih, kemudian memanggil *Export Class* (entity) yang sesuai. Export Class mengambil data dari *Database* (entity) sesuai filter, lalu melakukan proses *recursive* untuk memformat data menjadi file Excel (.xlsx) atau PDF. File yang telah dibuat dikembalikan ke ReportController, yang menginstruksikan Halaman Laporan untuk mengunduh file tersebut ke perangkat Pengguna.

Pada Gambar 4.X13 ditunjukkan *sequence diagram* untuk proses lihat dashboard. Interaksi dimulai ketika aktor *Pengguna* membuka halaman dashboard. Boundary *Halaman Dashboard* mengirimkan request ke control *DashboardController*. DashboardController melakukan proses *recursive* pertama untuk mengidentifikasi role pengguna yang sedang login, kemudian meminta *Model* (entity) untuk mengambil data statistik dari *Database* (entity) yang meliputi data absensi, nilai, siswa, dan guru. Setelah data diterima, DashboardController melakukan proses *recursive* kedua untuk menghitung ringkasan data seperti total guru hadir, persentase kehadiran, rata-rata nilai, dan jumlah siswa per tingkat. Hasilnya dikirim ke Halaman Dashboard yang menampilkan grafik, statistik, dan tabel sesuai dengan role pengguna.
